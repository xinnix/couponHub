## 系统概述

OpenCode 是一个优惠券管理销售系统,包含管理后台(Admin)和小程序端(Miniapp)。管理端负责商户管理、券模板创建、订单管理、结算处理等功能。

### 核心功能

- **商户管理**: 商户信息维护、核销员配置
- **券模板管理**: 创建/编辑优惠券,配置适用商户范围
- **订单管理**: 查看、退款、导出订单
- **结算管理**: 生成商户结算单、确认支付
- **新闻资讯**: 发布新闻、关联优惠券
- **权限管理**: RBAC 角色权限系统

## 用户体系

系统采用双用户认证体系,管理端和小程序端用户完全隔离。

### 1. Admin 用户(管理后台)

**身份**: 系统管理员、运营人员

**权限**: RBAC 角色-权限系统控制

**登录方式**: 邮箱 + 密码

**数据访问**: 可访问所有数据,无数据隔离

**Token 格式**:
```json
{
  "sub": "admin_id",
  "email": "admin@example.com",
  "type": "admin"
}
```

### 2. User 用户(小程序)

**身份**: 消费用户、商户核销员

**权限**: 基础用户权限(领券、核销)

**登录方式**: 微信登录 / 邶箱注册

**数据访问**: 只能访问自己的数据(`where: { userId: user.id }`)

**Token 格式**:
```json
{
  "sub": "user_id",
  "email": "user@example.com",
  "type": "user"
}
```

### ⚠️ 重要约束

- JWT Strategy 自动识别用户类型(`type`字段)
- 小程序用户无权访问管理端路由
- 核销员身份通过 `User.handlerId` 关联 `Handler` 表

## 核心业务流程

### 1. 优惠券购买流程

```
用户浏览券模板
  ↓
选择优惠券
  ↓
创建订单(状态: UNPAID)
  ↓
预扣库存
  ↓
┌─────────┐
│ 免费券? │
└─────────┘
  │
  ├── Yes → 自动标记为 PAID(isFreeOrder: true)
  │
  └── No → 调起微信支付
            ↓
          支付成功回调
            ↓
          更新订单为 PAID
            ↓
          计算过期时间(paidAt + validDays)
```

**关键逻辑**:

- **分布式锁**: 创建订单时使用 Redis 锁防止超卖
- **库存预扣**: 创建订单时立即扣减库存
- **领取限制**: 每人限领数量检查(`claimLimit`)
- **免费券**: `buyPrice = 0` 时自动支付,无需用户支付

### 2. 扫码核销流程

```
商户核销员扫描用户二维码
  ↓
解析二维码内容(orderId + 时效性验证)
  ↓
验证核销员身份(Handler.isActive)
  ↓
验证订单状态(必须为 PAID)
  ↓
验证订单有效期(expireAt 字段)
  ↓
验证商户权限(merchantScope 包含当前商户)
  ↓
更新订单状态为 REDEEMED
  ↓
记录核销信息(merchantId、handlerId、redeemedAt)
```

**权限验证**:

- 核销员必须属于券模板的适用商户范围
- 二维码时效性验证(防止过期二维码)
- 订单必须未过期(`expireAt`字段检查)

### 3. 退款流程

```
用户申请退款
  ↓
验证订单状态(必须为 PAID)
  ↓
验证订单未核销(redeemedAt 为空)
  ↓
验证订单未过期(使用 validUntil 字段)
  ↓
更新订单状态为 REFUNDING
  ↓
调用微信退款接口
  ↓
等待微信回调确认
  ↓
回调成功后更新为 REFUNDED
  ↓
记录退款信息(refundId、refundedAt)
```

**注意事项**:

- 退款前需检查订单是否已核销
- 已核销订单无法退款
- 已过期订单需通过"过期退款"流程处理(待开发)
- 退款状态转换: PAID → REFUNDING → REFUNDED

### 4. 过期订单处理

```
定时任务扫描过期订单
  ↓
查询 status=PAID 且 expireAt < now 的订单
  ↓
批量更新为 EXPIRED 状态
  ↓
释放库存(stock 回增)
```

**定时任务**:

- 每 5 分钟执行一次
- 仅处理 PAID 状态的过期订单
- UNPAID 状态订单有独立的过期处理(未支付订单)

## 订单状态流转

### 状态定义

| 状态      | 描述     | 可流转目标状态                       |
| --------- | -------- | ------------------------------------ |
| UNPAID    | 待支付   | → PAID, → EXPIRED                    |
| PAID      | 待使用   | → REDEEMED, → REFUNDING, → EXPIRED   |
| REDEEMED  | 已核销   | 终态,无法流转                        |
| REFUNDING | 退款中   | → REFUNDED                           |
| REFUNDED  | 已退款   | 终态,无法流转                        |
| EXPIRED   | 已过期   | 终态,无法流转                        |

### 状态流转图

```
┌─────────┐
│ UNPAID  │ ────────────┐
└─────────┘              │
  │                      │
  │ 支付成功             │ 未支付超时
  ▼                      ▼
┌─────────┐          ┌─────────┐
│  PAID   │          │ EXPIRED │
└─────────┘          └─────────┘
  │  │  │               (终态)
  │  │  │
  │  │  └── 订单有效期过期 ───→ ┌─────────┐
  │  │                        │ EXPIRED │
  │  │                        └─────────┘
  │  │
  │  └───── 申请退款 ───────→ ┌──────────┐
  │                          │REFUNDING │
  │                          └──────────┘
  │                              │
  │                              │ 退款成功
  │                              ▼
  │                          ┌──────────┐
  │                          │ REFUNDED │
  │                          └──────────┘
  │                              (终态)
  │
  └─ 扫码核销 ───────────────→ ┌──────────┐
                              │ REDEEMED │
                              └──────────┘
                                  (终态)
```

### 状态转换规则

| 当前状态 | 目标状态 | 触发条件                     | 操作人   |
| -------- | -------- | ---------------------------- | -------- |
| UNPAID   | PAID     | 支付成功回调                 | 系统     |
| UNPAID   | EXPIRED  | 未支付超时(定时任务)         | 系统     |
| PAID     | REDEEMED | 商户扫码核销                 | 核销员   |
| PAID     | REFUNDING | 用户申请退款                 | 用户     |
| PAID     | EXPIRED  | 订单有效期过期(定时任务)     | 系统     |
| REFUNDING | REFUNDED | 微信退款回调确认             | 系统     |

### 关键字段说明

**Order 表核心字段**:

| 字段            | 说明                                       |
| --------------- | ------------------------------------------ |
| status          | 订单状态(6 种状态)                         |
| paidAt          | 支付时间(PAID 状态时记录)                  |
| expireAt        | 过期时间(paidAt + validDays 天)            |
| redeemedAt      | 核销时间(REDEEMED 状态时记录)              |
| refundId        | 微信退款单号(REFUNDING/REFUNDED 状态)      |
| refundedAt      | 退款完成时间(REFUNDED 状态时记录)          |
| isFreeOrder     | 是否免费领取订单(buyPrice = 0 时标记)      |
| redeemMerchantId | 核销商户 ID(REDEEMED 状态时记录)           |
| handlerId       | 核销员 ID(REDEEMED 状态时记录)             |

## 模块使用指南

### 1. 商户类别管理

**功能**: 管理商户分类(餐饮、娱乐、零售等)

**操作流程**:

1. 创建类别(名称、slug、图标、排序)
2. 设置类别状态(ACTIVE/INACTIVE)
3. 配置排序权重(sortOrder,越小越靠前)

**使用场景**:

- 券模板可关联类别(筛选条件)
- 商户必须属于某个类别
- 小程序端按类别展示商户列表

### 2. 商户管理

**功能**: 维护商户信息、配置核销员

**核心字段**:

- 基本信息: 名称、logo、类别、区域、楼层、电话
- 状态控制: ACTIVE/INACTIVE
- 图片展示: gallery(JSON 数组,多张图片)

**核销员配置**:

1. 添加核销员(姓名、手机号)
2. 核销员手机号匹配 User 表(自动关联)
3. 设置核销员状态(isActive)

**权限控制**:

- 核销员只能核销适用商户范围内的券
- `CouponTemplate.merchantScope` 数组包含商户 ID

### 3. 券模板管理

**功能**: 创建/编辑优惠券模板

**核心配置**:

| 字段          | 说明                                       | 注意事项                      |
| ------------- | ------------------------------------------ | ----------------------------- |
| buyPrice      | 购买价格                                   | 0 表示免费券                  |
| faceValue     | 面值                                       | 用户看到的价值                |
| settlementAmount | 结算金额                                | 为空时使用 faceValue          |
| stock         | 库存                                       | 创建订单时自动扣减            |
| claimLimit    | 每人限领数量                               | null=无限制,默认 1 张         |
| validFrom/validUntil | 有效期范围                         | 创建订单时检查                |
| validDays     | 购买后有效天数                             | paidAt + validDays = expireAt |
| merchantScope | 适用商户范围(JSON 数组)                    | 核销权限验证                  |
| categoryId    | 商户类别(可选)                             | 用于筛选和分类                |
| featuredOnHome | 首页展示标记                              | 小程序首页超值优惠列表        |

**适用商户范围配置**:

1. 选择商户类别(餐饮、娱乐等)
2. 选择具体商户(多选)
3. 生成商户 ID 数组(JSON 格式)

**小程序码生成**:

- 系统自动生成小程序码图片(qrcodeUrl)
- 用户扫码直达领券页面
- 有效期 5 分钟(二维码时效性)

**状态控制**:

- ACTIVE: 正常销售
- EXPIRED: 已过期(系统自动标记)
- DISABLED: 手动禁用

### 4. 订单管理

**功能**: 查看、退款、导出订单

**查询筛选**:

- 用户信息(昵称、手机号)
- 券模板标题
- 订单状态(6 种状态筛选)
- 时间范围(创建时间、支付时间、核销时间)
- 商户信息(核销商户)

**订单详情**:

- 基本信息: 订单号、价格、面值、状态
- 用户信息: 昵称、手机号
- 券模板信息: 标题、有效期
- 时间信息: 创建时间、支付时间、过期时间、核销时间
- 核销信息: 核销商户、核销员姓名

**退款操作**:

1. 查看订单详情
2. 点击"申请退款"
3. 填写退款原因
4. 系统调用微信退款接口
5. 等待微信回调确认
6. 订单状态更新为 REFUNDED

**批量退款**:

- 选择多个 PAID 状态订单
- 批量申请退款
- 适用于大规模退款场景

**导出功能**:

- 导出订单列表(Excel 格式)
- 包含订单详细信息
- 用于财务报表、数据分析

### 5. 结算管理

**功能**: 生成商户结算单、确认支付

**结算流程**:

1. 定时任务生成月度结算单(每月初)
2. 统计上月核销订单(REDEEMED 状态)
3. 计算结算金额(sum(settlementAmount or faceValue))
4. 生成结算单(状态: PENDING)
5. Admin 确认结算单(状态: CONFIRMED)
6. Admin 标记已支付(状态: PAID)

**结算单内容**:

- 基本信息: 商户名称、结算周期、订单数量
- 金额信息: 总结算金额
- 状态流转: PENDING → CONFIRMED → PAID
- 快照数据: JSON 格式订单详情(审计证据)

**数据隔离**:

- 商户只能查看自己的结算单
- Admin 可查看所有结算单

### 6. 新闻资讯管理

**功能**: 发布新闻、关联优惠券

**核心配置**:

| 字段     | 说明                  | 使用场景                     |
| -------- | --------------------- | ---------------------------- |
| title    | 标题                  | 新闻标题                     |
| bannerUrl | 头图 URL             | 新闻列表展示图片             |
| content  | 富文本内容            | 新闻详情                     |
| status   | DRAFT/PUBLISHED       | 草稿/已发布                  |
| isHero   | 是否头图文章          | 小程序顶部展示               |
| isPopup  | 是否首页弹窗新闻      | 小程序首页弹窗展示           |

**优惠券关联**:

- 新闻可关联多个优惠券
- 用户浏览新闻时查看相关优惠券
- 关联表: `NewsCouponRelation`(多对多)

**展示逻辑**:

- 小程序首页展示头图文章(isHero=true)
- 小程序首页弹窗展示弹窗新闻(isPopup=true)
- 新闻列表展示所有 PUBLISHED 状态新闻

## 权限控制

### RBAC 权限系统

**模型结构**:

```
Admin ── AdminRole ── Role ── RolePermission ── Permission
```

**权限定义**:

| Resource    | Action       | 说明              |
| ----------- | ------------ | ----------------- |
| merchant    | create       | 创建商户          |
| merchant    | read         | 查看商户          |
| merchant    | update       | 编辑商户          |
| merchant    | delete       | 删除商户          |
| coupon      | create       | 创建券模板        |
| coupon      | read         | 查看券模板        |
| coupon      | update       | 编辑券模板        |
| coupon      | delete       | 删除券模板        |
| order       | read         | 查看订单          |
| order       | refund       | 退款操作          |
| settlement  | read         | 查看结算单        |
| settlement  | confirm      | 确认结算单        |
| settlement  | pay          | 标记已支付        |

**角色配置**:

| Role         | Permissions          | 说明              |
| ------------ | -------------------- | ----------------- |
| superadmin   | 所有权限             | 超级管理员        |
| operator     | merchant, coupon, order | 运营人员        |

**权限检查**:

- Admin 登录时获取角色权限列表
- 每次请求检查权限(`resource:action`)
- 无权限返回 403 Forbidden

## 操作注意事项

### 1. 数据一致性

**库存管理**:

- ⚠️ 创建订单时立即扣减库存(预扣)
- ⚠️ 退款成功后需手动释放库存(待开发)
- ⚠️ 过期订单自动释放库存(定时任务)

**领取限制**:

- ⚠️ `claimLimit` 检查只统计 PAID 状态订单
- ⚠️ 免费券和付费券共享领取限制
- ⚠️ 已退款订单不计入领取限制

### 2. 时效性控制

**订单有效期**:

- ⚠️ 有效期计算: `paidAt + validDays 天`
- ⚠️ 过期时间存储在 `expireAt` 字段
- ⚠️ 核销时检查 `expireAt` 字段(不是 `validUntil`)

**券模板有效期**:

- ⚠️ `validFrom` 和 `validUntil` 控制券模板的销售期
- ⚠️ 过期的券模板无法创建新订单
- ⚠️ 已购买的订单不受模板有效期影响(使用 `expireAt`)

**二维码时效性**:

- ⚠️ 小程序码有效期 5 分钟
- ⚠️ 过期二维码无法核销
- ⚠️ 用户需重新生成二维码

### 3. 状态转换

**终态订单**:

- ⚠️ REDEEMED、REFUNDED、EXPIRED 状态无法再次流转
- ⚠️ 状态转换遵循状态机规则
- ⚠️ 非法状态转换返回 400 错误

**退款限制**:

- ⚠️ 已核销订单无法退款
- ⚠️ 已过期订单需使用"过期退款"流程(待开发)
- ⚠️ REFUNDING 状态订单等待回调确认

### 4. 权限验证

**核销权限**:

- ⚠️ 核销员必须属于券模板的适用商户范围
- ⚠️ `merchantScope` 数组包含商户 ID
- ⚠️ 非适用商户核销员扫码返回 403 错误

**数据隔离**:

- ⚠️ 小程序用户只能访问自己的数据
- ⚠️ Admin 用户可访问所有数据
- ⚠️ 商户只能查看自己的结算单和核销记录

### 5. 定时任务

**订单过期处理**:

- ⚠️ 每 5 分钟扫描一次过期订单
- ⚠️ 仅处理 PAID 状态订单(未支付订单独立处理)
- ⚠️ 过期订单状态更新为 EXPIRED

**券模板过期处理**:

- ⚠️ 定时任务扫描 `validUntil < now` 的券模板
- ⚠️ 自动更新状态为 EXPIRED
- ⚠️ 不影响已购买的订单

**结算单生成**:

- ⚠️ 每月 1 日自动生成上月结算单
- ⚠️ 统计上月 REDEEMED 状态订单
- ⚠️ 生成快照数据(JSON 格式)
